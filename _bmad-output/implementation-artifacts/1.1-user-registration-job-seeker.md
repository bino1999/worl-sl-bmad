**Complexity:** Medium

## Story Requirements

### User Story

**As a** job seeker,
**I want** to register with my details,
**so that** I can access job opportunities on the platform.

### Acceptance Criteria

1. **Registration Form Components**
   - Form displays all required fields: full name, mobile number, email, password, location (district/city)
   - Form includes role selection (pre-selected as "Job Seeker" but visible)
   - Form has field validation with real-time feedback
   - Form submits only when all required fields are valid

2. **Input Validation**
   - Full name: non-empty, minimum 2 characters
   - Mobile number: valid Sri Lankan format (accepted formats: +94XXXXXXXXX or 0XXXXXXXXX)
   - Email: valid email format, no leading/trailing spaces
   - Password: minimum 8 characters, must include uppercase, lowercase, and number
   - Location: selected from dropdown or search of Sri Lankan districts/cities
   - Error messages shown immediately when field validation fails

3. **Email Verification**
   - On successful form submission, system sends verification email to provided address
   - Email contains verification link (token-based, 24-hour expiry)
   - User cannot log in until email is verified
   - Verification page shows success message upon completion
   - If link expires, user can request a new verification email

4. **Account Creation & Data Persistence**
   - User account is created in database with status "unverified"
   - Password is hashed using bcrypt before storage
   - Email is stored in normalized form (lowercase, trimmed)
   - Mobile number is stored in normalized form
   - Account activation timestamp is recorded upon email verification
   - Session is NOT created until email verification is complete

5. **User Experience & Feedback**
   - Loading indicator appears during form submission
   - Successful submission redirects to verification instruction page
   - Verification instruction page displays registered email (partially masked for privacy)
   - Error messages are specific and actionable (not generic "error occurred")
   - Resend verification email link is prominently available
   - User is informed of expected verification email arrival time (within 1 minute)

6. **Security Considerations**
   - Password reset link is invalid if already used
   - Rate limiting on registration endpoint (max 5 requests per IP per minute)
   - Rate limiting on verification email resend (max 3 requests per email per hour)
   - No existing user detection (prevent email enumeration attacks)
   - HTTPS encryption for all data transmissions

7. **Duplicate Prevention**
   - If email already exists, show friendly message: "This email is already registered. Please log in or use a different email."
   - If mobile number already exists by same user, silently continue registration (mobile uniqueness per user allowed for potential future features)

8. **Mobile Responsiveness**
   - Form is fully functional on screens 320px and wider
   - Touch-friendly button sizes (min 44px × 44px)
   - Input fields are appropriately sized for mobile keyboards
   - Error messages do not overlap form fields

### Database Schema Requirements

**Source:** [architecture.md - Database section; prd.md - FR-001, FR-002]

```sql
-- users table
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  mobile_number VARCHAR(20) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  full_name VARCHAR(255) NOT NULL,
  role ENUM('job_seeker', 'employer') NOT NULL,
  location VARCHAR(100) NOT NULL,
  status ENUM('unverified', 'active', 'suspended') DEFAULT 'unverified',
  email_verified_at TIMESTAMP,
  profile_photo_url VARCHAR(500),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  deleted_at TIMESTAMP
);

-- email_verifications table (for tracking verification tokens)
CREATE TABLE email_verifications (
  id SERIAL PRIMARY KEY,
  user_id INTEGER UNIQUE NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  verification_token VARCHAR(255) UNIQUE NOT NULL,
  token_expires_at TIMESTAMP NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- audit_log table (optional, for security compliance)
CREATE TABLE audit_logs (
  id SERIAL PRIMARY KEY,
  user_id INTEGER REFERENCES users(id),
  action VARCHAR(50) NOT NULL,
  ip_address VARCHAR(45),
  user_agent TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### API Endpoint Specification

**Source:** [prd.md - FR-001 User Registration, FR-002 Profile Management]

**Endpoint: POST /api/auth/register**

**Request Body:**
```json
{
  "fullName": "Janith Silva",
  "mobileNumber": "+94771234567",
  "email": "janith@example.com",
  "password": "SecurePass123",
  "location": "Colombo"
}
```

**Response (201 Created):**
```json
{
  "success": true,
  "message": "Registration successful. Please verify your email.",
  "data": {
    "userId": 42,
    "email": "janith@example.com",
    "role": "job_seeker",
    "status": "unverified"
  }
}
```

**Response (400 Bad Request - Validation Error):**
```json
{
  "success": false,
  "message": "Validation failed",
  "errors": {
    "email": "Invalid email format",
    "password": "Password must contain uppercase, lowercase, and number"
  }
}
```

**Response (409 Conflict - Email exists):**
```json
{
  "success": false,
  "message": "This email is already registered. Please log in or use a different email."
}
```

**Endpoint: POST /api/auth/verify-email**

**Request Body:**
```json
{
  "token": "verification_token_from_email"
}
```

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Email verified successfully. You can now log in.",
  "data": {
    "userId": 42,
    "email": "janith@example.com",
    "status": "active"
  }
}
```

## Architecture Compliance Notes

**Source:** [architecture.md - Authentication, Security & Compliance]

### Authentication Architecture Decisions

1. **JWT-Based Sessions:** While registration doesn't create a session immediately, the backend must be prepared to issue JWT tokens after email verification. The JWT should include:
   - `userId`: User's database ID
   - `email`: User's email (hashed or salted in token for security)
   - `role`: 'job_seeker' (for this story)
   - `iat` (issued at): Timestamp
   - `exp` (expiry): 24 hours from issue
   - `iss` (issuer): Application name for validation

2. **Password Hashing:** Use bcrypt with cost factor ≥ 10. Example in Node.js:
   ```javascript
   const hashedPassword = await bcrypt.hash(password, 10);
   ```

3. **Email Service Integration:** Use transactional email service (SendGrid, Mailgun). Registration must not block on email delivery; queue async job if possible.

4. **GDPR Alignment:** 
   - Collect only necessary personal data (full name, email, mobile, location)
   - Provide clear data usage policy in registration form
   - Ensure user can request data deletion (implement later in Profile Management epic)

## Latest Technical Information

**Source:** Web research as of March 18, 2026

### React 19 & TypeScript 5.4 Updates
- Use React Server Components sparingly; not needed for registration form
- TypeScript 5.4 supports const type parameters — use for generic form validation
- Material-UI v6.0 or Chakra UI v3.0 recommended for best React 19 support

### Express.js & bcryptjs Current Practices
- bcryptjs v2.4.3 is stable; bcrypt v5.x not yet production-recommended
- Express.js v4.18+ supports native async error handling without try-catch wrappers
- Use helmet.js for HTTP security headers (recommended version: 7.1.0+)

### PostgreSQL Best Practices (2026)
- ENUM types are acceptable for role/status columns in PostgreSQL 14+
- Consider partitioning users table if expecting 1M+ users (not needed for v1 launch)
- Use UNIQUE constraints on email to prevent duplicates at database level

### Email Security Standards
- SPF, DKIM, DMARC configuration required for production email sending
- Verification token should be cryptographically random (use crypto.randomBytes)
- Token length: minimum 32 bytes (256 bits) recommended
